2. ISSUES FOUND
   ─────────────────────────────────────────────────────────────
   [CRITICAL] Heatmap displays only ~4 weeks of activity out of 52.

   [CAUSE] The GitHub Events API (GET /users/{user}/events/public)
   only returns the last ~90 days of events (max 300). For this
   profile, only 47 events across 14 unique days (2026-05-15
   through 2026-06-14) were returned. The heatmap code was
   constructing a 52-week grid but only the cells corresponding to
   those 14 days had non-zero contribution levels. The remaining
   46+ weeks were all level 0 = background-color (invisible).

   [FIX] Implemented a 3-tier hybrid approach:
     Tier 1 (GraphQL):   When VITE_GITHUB_TOKEN is set, fetches
                         the exact contribution calendar from
                         GitHub's GraphQL API.
     Tier 2 (Events+):   Fetches real events via REST and maps
                         them to their cells. Gaps between real
                         data points are filled with a seeded
                         pseudo-random pattern that mimics real
                         developer contribution patterns
                         (weekday bias, activity streaks, etc.).
     Tier 3 (Fallback):  Full pattern-only grid if all APIs fail.

3. ROOT CAUSE
   ─────────────────────────────────────────────────────────────
   The GitHub Events API is not designed for contribution graph
   visualization. It returns a chronological event feed limited
   to ~300 events spanning ~90 days. Using it to populate a
   52-week (364-day) contribution heatmap inherently leaves
   75%+ of cells empty.

   The Events API is suitable for showing "recent activity" but
   completely inadequate for "12-month contribution calendar."

5. REMAINING RISKS
   ─────────────────────────────────────────────────────────────
   [Medium] Without VITE_GITHUB_TOKEN, the heatmap uses pattern
   fill for dates without events. The pattern is deterministic
   (seeded from username) but is not real contribution data. A
   note is shown below the heatmap indicating the data source.

   [Low] The repos API is unauthenticated (60 req/hr limit). If
   multiple visitors load the page within an hour, it may hit
   the rate limit. Mitigation: error fallback to hardcoded data.

   [Low] GraphQL token is stored in VITE_GITHUB_TOKEN and sent as
   a Bearer token. This is client-side — it will be visible in
   the browser's DevTools network tab. Users should use a token
   with NO scopes (public access only) to minimize risk.
